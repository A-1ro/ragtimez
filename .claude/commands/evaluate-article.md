# 記事評価

最新の自動生成記事に対して、ファクトチェック・品質評価・プロンプト改善提案の3軸で評価を行う。

## 引数

`$ARGUMENTS` で対象を指定可能:
- `/evaluate-article` — 最新の日本語記事を評価
- `/evaluate-article 2026-04-12` — 指定日付の日本語記事を評価
- `/evaluate-article en` — 最新の英語記事を評価
- `/evaluate-article 2026-04-12 en` — 指定日付の英語記事を評価

## 手順

### 1. 対象記事の特定

`$ARGUMENTS` をパースして対象記事を決定する:
- 日付指定がなければ `src/content/articles/` 内の最新日付のファイルを使用
- `en` 指定があれば `.en.md`、なければ `.md` を対象とする
- 対象ファイルを Read で読み込む

### 2. 生成プロンプトの確認

`src/pages/api/generate-article.ts` から、記事生成に使われているプロンプト（Step 0〜2）の構造を把握する。評価フェーズ 3 のプロンプト改善提案で参照する。

### 3. 評価の実行

以下の3フェーズを順に実行し、それぞれの結果を記録する。

---

#### フェーズ 1: ファクトチェック

記事内の事実主張を検証する。WebSearch を使って裏取りを行う。

**チェック項目:**

| # | 項目 | 説明 |
|---|------|------|
| F1 | ソース URL の実在性 | frontmatter の `sources[].url` が実在するか（WebSearch でタイトル・URL を検索） |
| F2 | バージョン番号・日付 | 記事中に登場する具体的なバージョン番号、リリース日、数値が正確か |
| F3 | API 名・パラメータ名 | 記事中の API 名、関数名、パラメータ名が公式ドキュメントと一致するか |
| F4 | ベンチマーク・数値 | 性能数値や比較データがソースに裏付けられているか |
| F5 | 因果関係・帰属 | 「X が Y を発表した」「X の理由は Y」などの主張がソースと一致するか |
| F6 | ハルシネーション検出 | ソースに記載のない詳細（架空の機能、存在しない API）が捏造されていないか |

**検証手順:**
1. 記事本文からすべての事実主張（固有名詞、数値、技術的主張）を抽出する
2. frontmatter の sources URL をそれぞれ WebSearch で検証する
3. 主要な技術的主張を WebSearch で裏取りする
4. 各主張に対して: **Verified** / **Unverified** / **Incorrect** / **Fabricated** のラベルを付与する

---

#### フェーズ 2: 記事品質評価

記事の構造・内容・表現の品質を多角的に評価する。

**チェック項目:**

| # | 項目 | 配点 | 評価基準 |
|---|------|------|----------|
| Q1 | 構造遵守 | /15 | 3〜5 セクション、## 見出し、最終セクションが まとめ/Summary、段落 2〜3 文 |
| Q2 | 技術的深度 | /20 | 表面的な説明ではなく、仕組み・設計判断・トレードオフに踏み込んでいるか |
| Q3 | 具体性 | /20 | バージョン番号、API 名、コード例、ベンチマークなど具体的な情報があるか |
| Q4 | 冗長性・反復 | /15 | セクション間で同じ情報を繰り返していないか。フィラー表現がないか |
| Q5 | まとめの実用性 | /10 | まとめのバレットポイントがエンジニアにとってアクション可能か |
| Q6 | ソース活用度 | /10 | 提供されたソースの情報をどの程度活用しているか（vs 一般論） |
| Q7 | 言語品質 | /10 | 日本語/英語として自然か。機械翻訳調の不自然な表現がないか |

**評価手順:**
1. 各項目を 0〜配点上限 で採点する
2. 合計 /100 点を算出する
3. 特に問題のある箇所を具体的に引用して指摘する

**グレード:**
- S: 90〜100（出版品質）
- A: 75〜89（良好、軽微な改善余地）
- B: 60〜74（及第点、明確な改善ポイントあり）
- C: 45〜59（要改善、複数の問題）
- D: 0〜44（不合格、大幅な改善が必要）

---

#### フェーズ 3: プロンプト改善提案

フェーズ 1・2 で発見された問題に基づき、`generate-article.ts` のプロンプトに対する具体的な改善案を提案する。

**分析対象:**
- Step 0（トピック選定）のシステムプロンプト
- Step 1（メタデータ生成）のシステムプロンプト
- Step 2（本文生成）のシステムプロンプト
- パイプライン全体の構造（Step 分割、temperature、max_tokens）

**提案の形式:**
各提案は以下の構造で記述する:

```
### P{番号}: {提案タイトル}

**対象:** Step {0|1|2} / パイプライン全体
**問題:** フェーズ 1/2 で発見された具体的な問題の説明
**現状のプロンプト:**
> （該当箇所の引用）

**改善案:**
> （修正後のプロンプト文言）

**期待される効果:** 改善によって解決される問題の説明
```

**提案すべき改善の例:**
- ハルシネーション防止の指示強化
- 具体性向上のための指示追加
- 冗長性抑制のための制約追加
- 構造遵守を強化する指示の明確化
- temperature / max_tokens の調整提案
- ソース情報の活用を促す指示の追加

## 注意事項

- WebSearch が使えない場合でも、記事本文とソース情報だけで可能な範囲のファクトチェックを実施する
- ファクトチェックでは「ソースに書かれていない詳細の捏造」を最も重要な問題として扱う（Llama 3.3 のハルシネーション傾向を意識する）
- プロンプト改善提案は実装可能な具体的レベルで記述し、抽象的なアドバイスに留めない
- 評価は厳格に行う。「良い記事です」で終わらせず、必ず改善ポイントを見つける
